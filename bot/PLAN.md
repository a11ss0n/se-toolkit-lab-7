# План разработки Telegram-бота для LMS

## Обзор

Данный документ описывает план разработки Telegram-бота для взаимодействия с LMS backend. Ключевая архитектурная особенность — testable handlers: логика команд отделена от Telegram-specific кода и может тестироваться независимо через CLI.

## 1. Scaffold — Структура проекта

Создаём модульную архитектуру с чётким разделением ответственности:

- **bot.py** — entry point с поддержкой двух режимов: Telegram bot (production) и CLI test mode (разработка/тестирование)
- **handlers/** — модули с обработчиками команд. Каждый хендлер — это чистая функция, принимающая контекст и аргументы, возвращающая текст ответа. Не зависит от aiogram.
- **services/** — клиенты внешних API (LMS backend, LLM). Инкапсулируют HTTP-запросы и логику повторных попыток.
- **config.py** — централизованная загрузка переменных окружения через pydantic-settings.

Такая структура позволяет юнит-тестировать бизнес-логику без моков Telegram и запускать ручные тесты через `uv run bot.py --test "/command"`.

## 2. Backend Integration — Slash команды с реальными данными

Реализуем базовые slash-команды, которые взаимодействуют с LMS API:

- **/start** — приветственное сообщение, краткая инструкция по использованию
- **/help** — полный список доступных команд с описанием
- **/health** — проверка статуса backend (вызов GET /health или аналогичного endpoint)
- **/labs** — получение списка лабораторных работ через LMS API (GET /items с фильтром по типу "lab")

Каждая команда использует сервисный слой для запросов к backend. Ошибки API обрабатываются gracefully — пользователь получает понятное сообщение вместо stack trace.

## 3. Intent Routing — LLM для понимания естественного языка

Вместо жесткой маршрутизации по командам реализуем интеллектуальный роутинг:

- Пользователь пишет естественным языком: "покажи мои лабораторные" или "как сдать проект?"
- Сообщение отправляется в LLM через LLM API с системным промптом для классификации интентов
- LLM возвращает название команды и извлечённые параметры (slot filling)
- Бот вызывает соответствующий хендлер с распознанными параметрами

Это позволяет создать более естественный UX без необходимости запоминать команды.

## 4. Deployment — Контейнеризация и развертывание

Финальный этап — подготовка к production:

- **Dockerfile** — многоэтапная сборка с uv для установки зависимостей, запуск через `uv run bot.py`
- **docker-compose.yml** — сервис бота с переменными окружения из .env.bot.secret
- **Health checks** — endpoint для проверки работоспособности бота
- **Logging** — структурированные логи с уровнями INFO/ERROR для мониторинга
- **Graceful shutdown** — корректная остановка при получении SIGTERM

Развертывание выполняется вместе с остальными сервисами LMS через docker-compose up.

## Критерии готовности

- [ ] Все slash-команды работают в test mode
- [ ] Интеграция с LMS API проверена на реальных данных
- [ ] LLM intent routing классифицирует >= 5 типов запросов
- [ ] Docker-образ собирается и запускается без ошибок
- [ ] Покрытие тестами handlers >= 80%
